## 1. 项目定位与参赛价值

### 1.1 项目名称

**Cassandra: 面向生物医药尽职调查的多智能体取证分析平台**

### 1.2 一句话定义

Cassandra 是一个围绕生物医学论文、临床试验、补充材料和科研图像展开的专业多智能体分析系统，目标是为投资研究、科研诚信审查和药物风险评估提供可追踪、可解释、可落地的证据链与分析报告。

### 1.3 参赛价值

与通用型 Agent 不同，Cassandra 的价值不在于“会对话”，而在于“会完成高风险专业任务”。它面向的不是开放域聊天场景，而是一个高门槛、强证据依赖、结果容错率极低的垂直场景：生物医药研究尽调。

竞赛视角下，Cassandra 具备以下优势：

- 任务目标清晰：输入一个药物、疗法或研究问题，输出完整的风险研判报告。
- Agent 分工明确：采集、文本审计、图像取证、报告生成职责分离。
- 数据流闭环完整：从外部检索、全文下载、证据抽取，到报告生成与图谱落盘形成端到端流水线。
- 过程可观察：前端实时展示进度、日志、图谱节点和法证图片预览。
- 工程约束充分：对长文本、坏 PDF、模型输出异常、网络抖动都做了显式处理。

从比赛评委视角看，Cassandra 展示的是“多智能体系统如何真正落到复杂专业流程中”，而不是单点模型能力展示。

## 2. 业务问题定义

### 2.1 行业背景

生物医药研究存在非常严重的信息不对称。公开摘要、新闻稿和企业路演通常只呈现正面结果，而真正有价值的负面信号往往埋在以下位置：

- 补充材料中的不显著结果
- 临床试验中止、撤回、暂停记录
- 图像数据中的异常拼接或重复痕迹
- 方法学章节中的协议偏移和样本筛除
- 全文与摘要之间的不一致描述

传统人工尽调存在几个根本问题：

- 阅读成本高：需要同时处理数十篇全文 PDF、补充表格和图像。
- 检索碎片化：文献、试验、评论、图谱信息分散在不同数据源。
- 证据整合困难：文本风险、图像风险和临床试验风险往往无法统一到一套框架中。
- 结果难复核：分析过程常常不可追踪，只能看到最终结论。

### 2.2 Cassandra 要解决的核心问题

Cassandra 不是要替代研究员的所有判断，而是要先完成最费时间、最容易遗漏、最适合自动化的工作：

1. 自动发现相关论文与失败试验。
2. 自动下载和处理可得 PDF 全文。
3. 自动抽取隐藏在补充材料中的风险信号。
4. 自动检查论文图像中的可疑模式。
5. 自动聚合为结构化证据上下文。
6. 自动生成可解释、可复核的投资级报告。

因此，Cassandra 的设计核心不是“生成答案”，而是“构造证据驱动的分析链”。

## 3. 系统总体架构

### 3.1 总体拓扑

当前主干代码采用 Flask + SocketIO 作为应用入口，LangGraph 作为编排中枢，多个专业 Agent 作为执行节点，Neo4j/Redis/本地缓存作为状态与知识图谱存储层。

整体架构可以概括为：

```text
用户输入
  -> Flask / CLI 入口
  -> LangGraph StateGraph 编排
  -> BioHarvestEngine
  -> 并行执行 EvidenceEngine + ForensicEngine
  -> GraphBuilder 聚合验证
  -> ReportWriter 分章节生成报告
  -> Markdown / HTML / PDF / Graph 输出
```

### 3.2 主干工作流

当前实现中的编排路径是一个清晰的 DAG：

```text
START
  -> harvester
  -> miner
  -> graph_builder
  -> writer
  -> END

同时 harvester -> auditor -> graph_builder
```

这意味着：

- Harvester 是串行入口节点。
- Miner 和 Auditor 是并行分支。
- GraphBuilder 是并行结果的汇合点。
- Writer 是最终综合节点。

该设计兼顾了两个目标：

- 最大化并行效率
- 保持状态收敛和输出稳定

### 3.3 为什么选 LangGraph

很多 Agent 系统会陷入两种问题：

- 仅做顺序脚本调用，缺乏真正的多 Agent 编排能力。
- 仅做对话型协作，缺乏稳定状态管理和任务确定性。

Cassandra 使用 LangGraph 的原因在于它提供了一个适合工程化 Agent 流水线的抽象：

- 节点是专业能力单元。
- 边定义控制流与并行关系。
- 共享状态对象作为节点间统一数据面。
- 支持同步执行与流式执行两种模式。

这使得 Cassandra 更接近“可部署的 Agent 系统”，而不是“实验性质的 prompt 拼接器”。

## 4. Multi-Agent 架构设计

### 4.1 架构类型判断

从当前代码实现看，Cassandra 属于**功能分工型、共享状态驱动型、并行汇合型 Multi-Agent 系统**。

它不是自由对话式 Agent 网络，也不是完全自治的博弈型 Agent 社会，而是一种更适合生产环境的专业流水线型架构。

### 4.2 三层抽象

#### 第一层：编排层

编排层由 LangGraph StateGraph 负责，定义谁先执行、谁并行执行、谁负责收敛最终状态。

编排层解决的问题是：

- 多个 Agent 如何协同完成同一任务
- 并行结果如何汇合
- 错误和状态如何统一表达

#### 第二层：能力层

能力层由四类核心角色组成：

- BioHarvestEngine：信息检索与资料获取
- EvidenceEngine：全文文本风险抽取
- ForensicEngine：图像法证分析
- ReportWriter：证据整合与报告生成

每个 Agent 只解决一类问题，避免职责重叠。

#### 第三层：表现层

表现层通过 Flask + SocketIO 将内部执行过程可视化：

- 进度百分比
- 当前执行步骤
- 实时日志
- 图谱节点与关系变化
- 图像法证扫描预览

这使得系统不仅能给出结论，也能向用户展示“结论是怎样一步步形成的”。

### 4.3 为什么这种架构适合比赛

竞赛型 Agent 项目通常容易偏向“对话体验”或“单模型演示”。Cassandra 的优势在于它有完整任务链路，可以回答评委最关心的四个问题：

1. Agent 为什么需要多个，而不是一个大模型？
2. 各个 Agent 的职责边界是什么？
3. 它们之间如何交换结果和状态？
4. 整体系统如何保证结果质量？

在 Cassandra 中，这四个问题都有明确答案，而且答案落在真实代码结构中。

## 5. Agent 角色与职责边界

### 5.1 BioHarvestEngine：证据入口 Agent

BioHarvestEngine 是整个系统的第一责任节点，它决定后续分析是否有足够高质量的输入。

其核心职责包括：

- 根据用户查询生成面向不同数据库的检索词
- 在 EuroPMC 检索可直接访问 PDF 的开放文献
- 在 PubMed 做补充检索
- 在 ClinicalTrials.gov 检索失败试验和中止记录
- 将检索结果合并成统一 evidence candidate 结构
- 下载 PDF 到本地，形成后续分析的可消费输入

它的关键设计点不是“会搜”，而是“会优先搜到可下载全文的材料”。这使得下游 Agent 真正能做全文和补充材料分析，而不是只看摘要。

### 5.2 EvidenceEngine：全文暗数据挖掘 Agent

EvidenceEngine 是 Cassandra 技术含量最高的节点之一。

它处理的是文本层面的隐蔽风险信号，重点关注：

- 数据未展示
- 统计学不显著但临床可疑的结果
- 安全性信号被弱化描述
- 协议变更、样本剔除、终止原因
- 补充图表和附录中的负面证据

它的输出不是普通摘要，而是双层结构：

- paper_summary：对单篇研究的详细摘要
- risk_signals：结构化风险项列表

这种协议化输出，使得下游 Writer 能够直接消费高质量上下文，而不是再次清洗自然语言噪声。

### 5.3 ForensicEngine：多模态图像取证 Agent

ForensicEngine 负责处理另一条完全不同的风险通道：图像层面的科研可信度问题。

它的典型检测对象包括：

- Western blot 拼接或重复条带
- 图表数据点异常复制
- 不一致背景纹理
- 显微图中的克隆区域
- 图像质量异常或局部增强痕迹

它的作用不是替代人工法证，而是作为高效率初筛器，将可疑图像快速从大量论文中标记出来。

### 5.4 GraphBuilder：状态收敛与证据聚合节点

GraphBuilder 在当前系统里承担的是“中间层收敛器”的角色。它把文本证据、图像法证、失败文件、分析状态等数据整合为统一的下游上下文。

它的重要性在于：

- 把并行分支的输出统一收口
- 对状态完整性进行校验
- 为 Writer 准备一致的数据接口
- 为知识图谱落库提供可用结构

### 5.5 ReportWriter：投资级输出 Agent

Writer 不负责发现事实，它负责把前面多个 Agent 的结果综合成可读、可复核、结构化的最终报告。

当前实现采用 segmented generation，即分章节生成，而不是一次性长文生成。这样做的好处是：

- 降低单次生成失败风险
- 提高章节结构稳定性
- 允许章节级重试
- 更适合超长证据上下文

最终输出可以渲染为 Markdown、HTML 和 PDF，并可附带图表和文献图片。

## 6. Agent 间通信机制

### 6.1 当前实现：共享状态通信

当前主干代码中，Agent 间最核心的通信方式不是直接对话，而是**共享状态对象 AgentState**。

这意味着每个节点都通过“读状态、写状态”的方式协同：

- Harvester 写入 harvested_data 和 pdf_paths
- Miner 读取 pdf_paths，写入 text_evidence 和 compiled_evidence_text
- Auditor 读取 pdf_paths，写入 forensic_evidence
- GraphBuilder 读取前述字段并统一汇合
- Writer 读取聚合状态并生成最终报告

这是一个非常典型、非常工程化的通信设计。它的优势是：

- 结构清晰
- 可调试性强
- 无需复杂消息协议
- 并行合并规则明确

### 6.2 状态合并策略

AgentState 中不同字段采用不同合并策略：

- List 字段使用追加合并
- 标量字段使用 last-write-wins

例如：

- pdf_paths、harvested_data、text_evidence、forensic_evidence、errors 会累加
- user_query、compiled_evidence_text、analysis_status、recommendation、risk_score 会采用后写覆盖

这一设计对于并行节点非常关键，因为 miner 和 auditor 的结果需要安全汇合到同一个状态对象中。

### 6.3 当前实现：事件流通信

除了共享状态，系统还存在第二条重要通信链：后端到前端的 SocketIO 事件流。

这条链路不是 Agent 之间的认知协商，而是系统可观察性通道。它包括：

- progress 事件：通知当前阶段与百分比
- step 事件：通知步骤状态变化
- log 事件：推送实时日志
- graph_update 事件：推送图谱节点与关系更新
- analysis_complete 事件：推送任务完成结果

这使得前端能够实时展示 Agent 系统当前在做什么，而不是等任务结束后才看到结果。

### 6.4 需要特别说明的事实边界

在项目文档和历史分析材料中，多次出现 ForumEngine、主持人 Agent、forum.log 等概念。这些内容对理解未来演进很有价值，但并不是当前主干代码中的现役通信机制。

当前仓库事实是：

- 旧备份代码已经明确标注 ForumEngine 已移除。
- 当前主干实现没有活跃的 agent-to-agent 对话回路。
- 当前多 Agent 协作依赖共享状态和编排控制，而不是自然语言协商。

这个边界在比赛答辩中非常重要，因为它体现了项目团队对“真实实现”和“规划能力”的严格区分。

### 6.5 未来增强方向：从共享状态到反思式协同

如果继续演进，Cassandra 最自然的增强方向不是增加更多 Agent，而是在现有专业 Agent 之间加入反思式协调层，例如：

- 由主持节点读取 EvidenceEngine 和 ForensicEngine 的阶段性摘要
- 生成下一轮分析提示
- 将提示写回共享状态供下游节点参考

这样的增强会使 Cassandra 从“流水线式多 Agent”进一步演进为“带反思机制的协同式多 Agent”。

## 7. 端到端数据流设计

### 7.1 输入层

系统支持两种输入入口：

- Web API：适合前端界面交互和文件上传
- CLI：适合开发、调试和批量实验

输入内容包括：

- 用户查询字符串
- 可选的本地 PDF 文件

### 7.2 采集层数据流

Harvester 节点完成以下数据转换：

```text
用户查询
  -> LLM 生成检索词
  -> EuroPMC / PubMed / ClinicalTrials.gov 检索
  -> evidence candidates
  -> 下载 PDF
  -> harvested_data + pdf_paths
```

这里产出的两个关键对象是：

- harvested_data：论文或试验的结构化元数据
- pdf_paths：本地 PDF 路径列表

这两个字段构成后续分析的输入基座。

### 7.3 文本分析分支数据流

EvidenceEngine 消费 pdf_paths 后，会对每个 PDF 执行：

1. 文本抽取
2. 质量检查
3. token 估算
4. 长文本直接分析或分块分析
5. 风险信号抽取
6. 输出 paper_summary 与 risk_signals

随后 supervisor 将多篇 PDF 的结果拼接为 compiled_evidence_text。其格式大致是：

```text
=== EVIDENCE SOURCE: 文件名 ===
> FILE: 文件名
> SUMMARY: 单篇详细摘要
> RISK FINDINGS: 风险项 JSON
--------------------------------------------------
```

这个字段是整个数据流中最关键的桥。因为 Writer 真正依赖的不是原始论文元数据，而是这份经过抽取、格式化、聚合后的证据上下文。

### 7.4 图像分析分支数据流

ForensicEngine 同样从 pdf_paths 出发，但它的处理链不同：

```text
pdf_paths
  -> 提取图片
  -> 过滤过小或无意义图像
  -> Gemini Vision 分析
  -> ImageAuditResult 列表
  -> forensic_evidence
```

在此过程中，日志链还会实时把当前扫描图片名和预览路径发给前端，使用户能够看到系统正在分析哪张图。

### 7.5 汇合层数据流

GraphBuilder 和 Writer 接收到的不是“两个孤立结果”，而是一个已经汇合的状态对象，至少包含：

- harvested_data
- text_evidence
- forensic_evidence
- compiled_evidence_text
- failed_files
- forensic_failed_files
- total_files
- analysis_status

Writer 由此构建生成上下文，完成从“证据状态”到“叙述文档”的最终跃迁。

### 7.6 输出层数据流

最终输出包括几类：

- 报告 Markdown
- HTML/PDF 渲染结果
- WebSocket 完成事件
- Neo4j 异步写入
- 本地图谱 JSON 缓存

这意味着 Cassandra 的输出不是单一文件，而是一组彼此关联的分析资产。

## 8. 核心技术实现

### 8.1 数据源策略：为什么 EuroPMC 是主源

在生物医学场景中，最稀缺的不是题录信息，而是可消费的全文 PDF。Cassandra 在这一点上采取了实用主义策略：

- EuroPMC 作为主源，因为开放获取比例高，且容易直接拿到 PDF URL。
- PubMed 作为补充检索源，避免漏掉重要文献。
- ClinicalTrials.gov 作为失败试验与中止记录主源。

这种策略体现的是“先确保分析材料可得”，而不是“先追求检索覆盖面最大化”。

### 8.2 超长上下文利用：为什么 EvidenceEngine 有竞争力

EvidenceEngine 的能力来源不是简单“调用大模型”，而是将长上下文能力用于高密度证据分析。

它具备以下特征：

- 使用超长上下文窗口处理整篇论文与补充材料
- 在输入过大时自动切块处理
- 输出协议化 JSON，降低后处理复杂度
- 对异常 PDF 提供显式错误类别

这使得它相比“只读摘要的 Agent”有数量级上的能力差异。

### 8.3 结构化协议：为什么系统比单次 Prompt 更稳

Cassandra 多处采用结构化协议而不是自由文本接口，例如：

- 检索词生成返回 JSON
- 单篇论文挖掘结果返回 paper_summary + risk_signals
- 图像法证返回 status + confidence + findings
- 共享状态定义为 TypedDict

这类协议化设计的意义在于：

- 降低模型输出不稳定性
- 降低下游解析复杂度
- 方便做自动校验
- 便于记录和重放中间结果

### 8.4 分章节报告生成

Writer 节点当前采用 segmented generation，其逻辑是：

1. 定义章节列表
2. 分配字数预算
3. 逐章生成
4. 章节失败时局部兜底
5. 组装完整文档

这比单次长文本生成更符合比赛中对稳定性和可控性的要求，因为它：

- 更易定位问题章节
- 更易展示生成过程
- 更适合处理复杂上下文
- 更容易控制输出结构

### 8.5 智能上下文压缩的取舍

项目中引入过 Smart Context Builder，但当前主流程里对其做了保守处理，原因是过度压缩会损失关键信息，尤其是 PMC 标识、具体实验细节和引用上下文。

这一点非常值得在比赛中强调：团队不是盲目追求“压缩越多越好”，而是在系统真实运行中根据质量反馈调整策略，宁可保留原始证据文本，也不牺牲可引用性和可信度。

## 9. 实时交互与可视化机制

### 9.1 为什么前端实时性重要

多数 Agent 系统只在任务结束后展示结果，用户无法理解执行过程，也无法判断系统是否卡死。Cassandra 在这一点上做得更像专业控制台。

### 9.2 事件模型

系统通过 SocketIO 推送多种事件：

- progress：百分比进度
- step：阶段状态
- log：运行日志
- graph_update：图谱节点和关系变化
- analysis_complete：完成结果
- analysis_error：异常结果

### 9.3 断线重连机制

为了保证长任务场景下的前端可用性，系统为事件流设计了历史缓存和补发能力：

- 使用 event_history 环形缓冲记录最近事件
- 每条事件带有递增 id
- 客户端重连后可以请求补发丢失事件

这使得 Cassandra 在长时分析任务里具备更好的用户体验和工程完整度。

### 9.4 图像法证实时预览

这是系统一个很适合比赛展示的亮点：

- 法证节点分析图片时会输出扫描日志
- 日志处理器自动识别图片文件名
- 生成前端可访问的 image_path
- 前端日志面板同步展示缩略图

这使得评委能直观看到“多模态 Agent 正在处理真实科研图像”，而不是仅看一条文本提示。

## 10. 存储与持久化策略

### 10.1 多层存储设计

Cassandra 没有把所有责任压到单一数据库，而是采用分层存储：

- Neo4j：知识图谱与关系网络
- Redis：状态缓存
- 本地 JSON：任务图谱和降级缓存
- 本地文件系统：PDF、图像、报告、日志

### 10.2 为什么需要 Neo4j

Neo4j 的作用不是替代报告，而是保存实体关系和证据连接，例如：

- 作者与论文关系
- 论文与药物、靶点、试验的关联
- 风险信号与证据来源的映射

这样做有两个好处：

- 支持后续图谱可视化和查询
- 支持跨任务积累知识资产

### 10.3 为什么需要本地 JSON 缓存

知识图谱不应成为主流程的硬依赖。为避免数据库不可用时影响主任务，系统设计了本地任务图谱缓存作为降级方案。

这是一种面向实际部署的设计：

- 数据库可用时，写入 Neo4j
- 数据库不可用时，前端仍可从本地缓存展示图谱

### 10.4 为什么 Neo4j 写入异步化

图谱写入被放在任务完成后的 daemon 线程中执行，这样做可以保证：

- 数据库延迟不会阻塞前端完成事件
- 用户能更快拿到最终报告
- 图谱构建不会干扰主分析链路

这是一个典型的“把耗时的非核心副作用后置”的工程选择。

## 11. 鲁棒性、容错与反幻觉设计

### 11.1 PDF 质量异常处理

EvidenceEngine 对多种 PDF 异常做了显式分类，而不是统一报错后退出：

- 加密 PDF
- 扫描版 PDF
- 损坏 PDF
- 文本抽取过少
- 路径不存在

这样可以让系统对失败原因有可解释性，也方便后续统计失败模式。

### 11.2 模型输出协议校验

项目多处使用 StreamValidator 和数据校验逻辑来清洗模型输出，目标是防止以下问题：

- JSON 格式不完整
- 键名缺失
- 结构不符合预期
- 局部字段类型错误

在比赛答辩中，这一点非常重要，因为它说明 Cassandra 并不是“信任模型输出”，而是在模型之上建立了一层严格协议。

### 11.3 输入规模控制

长上下文是优势，但过大的输入会带来网络和稳定性问题。为此系统实现了：

- token 估算
- 安全上限
- chunking 回退
- 证据文本最小长度校验
- 查询关键词相关性检查

这些机制共同保证：

- 上下文既足够丰富
- 又不至于不可传输或不可控

### 11.4 法证分支的网络断路器

ForensicEngine 在连续网络失败达到阈值后会停止后续图片分析，避免任务在不稳定网络下无限放大失败成本。

这是一种典型的分布式系统思维在 Agent 工作流中的应用。

### 11.5 防幻觉核心策略

Cassandra 防幻觉的根本方法不是“让模型不要幻觉”，而是通过架构控制减少模型无依据发挥的空间：

- 上游尽量提供真实全文和原始风险项
- 中游将证据聚合为结构化上下文
- 下游 Writer 验证 compiled_evidence_text 是否足够长
- 章节生成采用分段方式，降低上下文漂移

从工程角度看，这是比单纯 prompt 约束更有效的防幻觉策略。

## 14. 答辩摘要

### 14.1 核心结论

Cassandra 的本质不是一个“能聊天的 Agent”，而是一套**围绕生物医药风险尽调构建的专业多智能体流水线系统**。

它通过：

- 多源检索
- 全文暗数据挖掘
- 图像法证分析
- 共享状态协同
- 分章节报告生成
- 实时可视化反馈

把一个原本依赖高水平研究员长时间人工完成的复杂任务，转化为一个可并行、可追踪、可落地的 Agent 系统。

### 14.2 适合比赛展示的最终表述

如果用一句最适合在比赛答辩中概括的话，可以表述为：

**Cassandra 展示了多智能体系统在高专业门槛、高证据密度、高风险容错要求场景中的工程化落地方式。它不是通过让多个 Agent 聊天来体现智能，而是通过让多个专业 Agent 在统一状态图中协同完成一条真实、复杂、可复核的尽职调查任务链。**

### 14.3 面向评委的强调点

答辩时建议重点强调以下三个关键词：

- **专业性**：系统解决的是生物医药尽调，不是泛问答。
- **完整性**：从检索到报告形成完整闭环。
- **工程性**：状态管理、容错、实时事件、结构化协议都已经落地。
